# OpenClaw plugin

Package: **`openclaw-plugin-hiveclaw`** (`packages/openclaw-plugin-hiveclaw`). Plugin id: **`hiveclaw`**.

## Install

From the monorepo root (after a normal `pnpm run build`, or the script builds the package):

```bash
pnpm run openclaw:plugin
```

That registers the plugin **once per user** on this machine; every `openclaw gateway` process shares it. **Do not** run `openclaw plugins install` on `./packages/openclaw-plugin-hiveclaw` directly — pnpm workspace symlinks fail OpenClaw’s install scan; use the staging script.

Secrets are passed via **`plugins.entries.hiveclaw.config`** in gateway JSON (use **`"${HIVECLAW_CHAIN_PRIVATE_KEY}"`** etc.); the plugin **does not read `process.env`** so OpenClaw’s installer stays happy.

Merge `examples/openclaw-plugins-hiveclaw.json` into your gateway config. OpenClaw defaults to **`tools.profile: "coding"`**, which does **not** include provider plugins—add the plugin id to **`tools.alsoAllow`** so agents can call **`hiveclaw_ping`**, **`remember_shared`**, etc.:

```json
"tools": {
  "profile": "coding",
  "alsoAllow": ["hiveclaw"]
}
```

Alternatively set **`"tools": { "profile": "full" }`** if you want all registered tools without an explicit allowlist (see [OpenClaw tool configuration](https://docs.openclaw.ai/gateway/config-tools)).

## Bundled skill

`openclaw.plugin.json` lists **`"skills": ["./skills"]`** (paths relative to the package root). When the plugin is enabled, OpenClaw loads those skill directories like any other [skills](https://docs.openclaw.ai/tools/skills) roots—same **`SKILL.md`** + YAML frontmatter format as workspace skills. This ships **`hiveclaw_memory`**, a longer operating guide than fits in tool descriptions. Plugin skills sit at **low precedence**: a skill with the **same name** in the agent workspace (or other higher-precedence roots) **overrides** the bundled copy.

## Configuration (`HiveclawPluginConfig`)

Set these under **`plugins.entries.hiveclaw.config`** in gateway JSON. OpenClaw resolves **`"${VAR}"`** references when loading config — use that for secrets (e.g. **`"chainPrivateKey": "${HIVECLAW_CHAIN_PRIVATE_KEY}"`**) so values come from the shell environment **without** the plugin bundle reading **`process.env`** (clean OpenClaw plugin install).

| Field | Purpose |
|-------|---------|
| `rpcUrl` | RPC endpoint (defaults match testnet if omitted) |
| `indexerUrl` | 0G indexer |
| `bootstrapContract` | Bootstrap address |
| `hiveRegistryContract` | HiveRegistry address |
| `storagePrivateKey` | Storage signer (required for uploads / storage smoke) |
| `chainPrivateKey` | Chain signer (required for memory tools) |
| `expectedChainId` | Expected chain |
| `defaultHiveKeyHex` | Default hive symmetric key (32-byte hex) |
| `hiveKeysJson` | JSON string: hive id → hex key |
| `defaultHiveId` | Default hive for tools that accept optional `hiveId` |
| `privateComputerBaseUrl` | Private Computer API base |
| `privateComputerApiKey` | Optional API key |

CLI and dashboard code paths still support **`loadHiveclawConfig()`** from **`hiveclaw-core/load-config`** (env-based); the OpenClaw plugin uses **gateway config only**.

## Tools

| Tool | Description |
|------|-------------|
| `hiveclaw_ping` | Chain + HiveRegistry + 0G storage smoke; use before memory ops. |
| `hiveclaw_list_my_hives` | Lists hive ids from `memberHives` for the plugin chain wallet. |
| `remember_shared` | Encrypts content, uploads to 0G, `commitMemory` under `shared/<segment>`. |
| `recall_shared` | Decrypts latest shared segment after registry lookup. |
| `remember_private` | Same as shared but under `private/<your-address>/<segment>`. |
| `recall_private` | Decrypts latest private lane segment for your wallet. |
| `summarize_memory` | Recalls segments, calls Private Computer summarization; optional commit to shared. |
| `hiveclaw_reflect` | Reflection loop: recall → summarize → encrypt/upload/commit to `shared/<summarySegment>`. |

Memory mutating tools require **chain** key, **storage** key, **hive** key resolution for the hive (via config or env), and registry contract — mirroring CLI requirements.

See [Configuration](../configuration/environment) and [CLI reference](../cli/reference) for the same semantics with different surfaces.
